# Project Intent

Dieses Repository dokumentiert und entwickelt ein Projekt rund um **Prompt Driven Development (PDD)**.

Der Fokus liegt nicht darauf, möglichst schnell Software zu produzieren.

Der Fokus liegt darauf, **Softwareentwicklung nachvollziehbar zu machen**.

Das Repository ist gleichzeitig:

- ein Entwicklungsprojekt
- eine Dokumentation
- ein Experiment
- und die Grundlage für ein Buch / Manifest.

---

# Das Ziel

Das langfristige Ergebnis dieses Projekts ist:

1. Ein verständlich dokumentierter Entwicklungsprozess für PDD.
2. Ein statisch veröffentlichter Text (Website / Buchstruktur).
3. Ein Repository, das den gesamten Entstehungsprozess nachvollziehbar macht.

Das Projekt soll zeigen:

> Software entsteht nicht nur durch Code.  
> Sie entsteht durch Entscheidungen.

---

# Was hier gebaut wird

Das Endprodukt ist **kein Framework**.

Das Endprodukt ist eine **Erklärung eines Systems**.

Das Repository enthält deshalb:

- Texte
- Entscheidungen
- Experimente
- Prompts
- und Code, wenn er notwendig ist.

Code ist ein Werkzeug.
Nicht das Ziel.

---

# Die zentrale Idee

Viele technische Bücher zeigen nur das fertige Ergebnis.

Der eigentliche Denkprozess bleibt unsichtbar.

Dieses Repository verfolgt deshalb einen anderen Ansatz:

Der **Weg wird dokumentiert**, nicht nur das Resultat.

Deshalb existieren:

- Worklog (Chronologie)
- Decisions (Architekturentscheidungen)
- Prompts (Inputs an KI)
- Code / Dokumentation (Umsetzung)

---

# Docs-first Entwicklung

Dieses Projekt folgt einem **docs-first Ansatz**.

Das bedeutet:

Ideen und Entscheidungen werden zuerst formuliert.

Implementierung folgt später.

Nicht umgekehrt.

---

# Rolle von KI

KI ist ein Werkzeug in diesem Projekt.

Aber sie ist **nicht der Autor**.

KI wird verwendet für:

- Analyse
- Strukturierung
- Generierung von Entwürfen
- Reflexion über Entscheidungen

Die Verantwortung für Entscheidungen bleibt beim Menschen.

---

# Veröffentlichungsform

Die Inhalte dieses Repositories werden später als **statische Website** veröffentlicht.

Minimaler technischer Ansatz:

```

Markdown → statisches HTML

```

Keine schweren Frameworks.

Der Fokus liegt auf Inhalt und Nachvollziehbarkeit.

---

# Arbeitsprinzipien

Das Projekt folgt einigen einfachen Regeln:

- KISS (Keep It Simple)
- YAGNI (You Aren't Gonna Need It)
- kleine Iterationen
- nachvollziehbare Entscheidungen
- keine Änderung ohne Trace im Worklog

---

# Was dieses Repository NICHT ist

Dieses Projekt ist nicht:

- ein KI-Coding-Experiment
- ein Framework
- ein Tooling-Showcase

Das Ziel ist **Verstehen**, nicht Beeindrucken.

---

# Erfolgskriterium

Das Projekt ist erfolgreich wenn jemand dieses Repository lesen kann und versteht:

- wie Entscheidungen getroffen wurden
- wie PDD funktioniert
- wie ein solcher Prozess praktisch umgesetzt werden kann

Nicht nur theoretisch.

Sondern anhand eines echten Projekts.

---

# Kurzfassung

Dieses Repository ist ein **offenes Labor für Softwareentwicklung**.

Der Code ist sichtbar.

Die Entscheidungen sind sichtbar.

Der Prozess ist sichtbar.
