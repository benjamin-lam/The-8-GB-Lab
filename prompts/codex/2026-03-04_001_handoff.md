# Prompt — Codex Handoff (workflow-aware)

**tool/model:** Codex  
**status:** used  
**date:** 2026-03-04  
**owner:** Benjamin Lam

## Purpose

Repository operativ fortführen: docs-first System + minimaler Publisher (Markdown → HTML), ohne Overengineering.

## Context

Dieses Repo ist docs-first. Worklog dokumentiert Chronologie, /decisions dokumentiert Trade-offs als Kanon.
Zusätzlich werden alle Codex-Prompts in /prompts archiviert.

## Inputs / Repo state assumptions

- /worklog existiert und hat einen Eintrag für 2026-03-04
- /decisions existiert mit ADR 0001 (Decisions-System) und ADR 0002 (Docs-/Worklog-first)
- /prompts existiert mit Template/README
- GitHub Pages Ziel: statisches HTML, Markdown als Quelle, ohne Jekyll

## Task

1) Repo inspizieren: Struktur + READMEs verstehen.
2) Erstelle eine Taskliste (5–10 Aufgaben, je 1–2h), die das Repo “benutzbar” macht:
    - Schreib-/Review-Workflow (Blueprint → Draft → Review → Ready)
    - minimaler Build-Prozess (Markdown → HTML), so klein wie möglich
3) Implementiere die ersten 2 Tasks direkt, inkl. Doku:
    - fehlende minimale Workflow-Dateien ergänzen
    - Build-Step als Stub oder minimal funktional (ok wenn erst “hello pipeline”)
4) Halte den Workflow ein:
    - Für jede nennenswerte Änderung: Worklog-Update (kurz)
    - Für echte Trade-offs: neue ADR (Decision)
    - Für diesen Prompt: Stelle sicher, dass diese Datei im Repo unter /prompts/... existiert (falls nicht, anlegen)

## Constraints / Non-goals

- Kein heroisches “alles bauen”.
- Keine schweren Frameworks.
- Kein “Marketing polishing”.
- KISS/YAGNI: minimal, boring, reproducible.

## Quality gates

- Jede Änderung ist nachvollziehbar über Worklog + ggf. Decision.
- Jede neue Datei hat einen kurzen Zweck (README oder Header-Kommentar).
- Build-Prozess muss lokal ausführbar sein (auch wenn minimal).

## Expected output

- Taskliste (kurz, konkret)
- Implementierte Changes (als Commit(s) oder Patch)
- Aktualisierte Doku: wie man den Workflow nutzt und wie build/publish gedacht ist
